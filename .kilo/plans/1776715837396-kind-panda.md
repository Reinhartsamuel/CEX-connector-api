# WorkerManager Refactor Plan (All 7 Exchanges)

## Objective
Refactor current per-exchange worker processes into a single `WorkerManager` process with pluggable adapters, while preserving existing reliability primitives already in codebase (Redis Streams, ws-ready handshake, reconcile patterns, structured logging/metrics baseline).

Scope confirmed: **all 7 exchanges** (`gate`, `okx`, `hyperliquid`, `tokocrypto`, `bitget`, `mexc`, `bitmart`).

---

## Current-State Findings (from repo)

- Workers are already stream-based (`ws-control:{exchange}` + `XREADGROUP`), not pub/sub.
- Executors already publish stream commands to per-exchange keys (`xadd ws-control:{exchange}`).
- PM2 currently runs API + 7 worker processes + separate `reconcile-cron`.
- Workers have per-user `ensureConnection/closeConnection` and exponential reconnect.
- Intentional-close guard is still missing in worker connection state; reconnect-on-close bug still applies.
- Logging uses project logger (`createLogger`) and metrics counters/histograms already exist.
- Reconciliation currently exists as standalone cron process + per-worker connect-time snapshot reconcile.

Implication: implementation should adapt the spec to stream-based control routing and existing 7-exchange architecture rather than reintroducing pub/sub assumptions.

---

## Delivery Strategy

Use a safe staged migration with compatibility period:
1. Build new WorkerManager + adapters behind current stream contracts.
2. Run in shadow mode / selective cutover.
3. Switch PM2 to single worker process.
4. Remove legacy worker entrypoints after stability window.

---

## Detailed Implementation Plan

### Phase 0 — Foundations + Compatibility (no behavior regressions)

1. **Create shared base contracts**
   - Add `src/workers/base/types.ts`:
     - `ExchangeId` union for 7 exchanges.
     - `ControlCommand` stream command type (`op`, `userId`, optional `contract`, optional exchange-specific fields).
     - `DLQEvent`, `AdapterHealth`, `WorkerHealthAggregate`.
   - Add `src/workers/base/ConnectionState.ts` with:
     - `ws`, `pingInterval`, `reconnectTimer`, `backoff`, `intentionalClose`, `lastEventAt`, `errorCount`, `extra`.

2. **Introduce BaseAdapter abstraction**
   - Add `src/workers/base/BaseAdapter.ts` with abstract lifecycle methods:
     - `ensureConnection`, `closeConnection`, `closeAllConnections`, `restoreConnections`, `handleControlCommand`, `replayEvent`, `getHealth`, `reconcileAll`.
   - Include shared helpers in base class (or companion util):
     - reconnect scheduling with capped backoff,
     - intentional-close-aware `onWsClose` helper,
     - ping lifecycle helper,
     - error accounting + `lastEventAt` updates.

3. **Control command normalization**
   - Define one stream envelope standard for manager routing:
     - Preferred: global `ws-control` stream with required `exchange` field.
     - Backward compatibility: accept legacy `ws-control:{exchange}` streams during transition.

4. **Executor compatibility plan**
   - Update all executors to always include `exchange` field in command fields.
   - During migration period, executors dual-write (config flag):
     - global `ws-control` + legacy `ws-control:{exchange}`.
   - Remove dual-write once cutover complete.

---

### Phase 1 — Adapter Extraction (7 exchanges)

For each worker (`gate`, `okx`, `hyperliquid`, `tokocrypto`, `bitget`, `mexc`, `bitmart`):

1. Create adapter files under `src/workers/adapters/`:
   - `GateAdapter.ts`, `OKXAdapter.ts`, `HyperliquidAdapter.ts`, `TokocryptoAdapter.ts`, `BitgetAdapter.ts`, `MexcAdapter.ts`, `BitmartAdapter.ts`.

2. Migrate existing worker logic into adapter class methods:
   - startup restore logic,
   - connection open/login/subscribe,
   - message parsing + event handlers,
   - reconcile-on-connect,
   - close and reconnect behavior.

3. Apply **intentional close fix** consistently:
   - Set `intentionalClose = true` before explicit `ws.close()`.
   - `onWsClose` skips reconnect when intentional.

4. Preserve existing exchange-specific behavior:
   - auth formats,
   - ping cadence,
   - contract/symbol handling,
   - event-to-trade state mapping,
   - exchange-specific credential requirements.

5. Add adapter-level health reporting:
   - active connection count,
   - `last_event_at`,
   - rolling `errors_last_5m`,
   - uptime.

---

### Phase 2 — WorkerManager Runtime

1. Add `src/workers/index.ts` as single entrypoint:
   - Build shared infra once (Redis clients, DB handle, logger, metrics).
   - Instantiate adapter registry (`Map<exchange, BaseAdapter>`).
   - Run `restoreConnections()` for all adapters in parallel with bounded concurrency.

2. Add `src/workers/ControlRouter.ts`:
   - Consume commands from global `ws-control` stream consumer group.
   - Route by `exchange` to correct adapter.
   - Fallback compatibility readers for legacy `ws-control:{exchange}` streams during transition.
   - ACK semantics and poison-message logging.

3. Add `src/workers/ReconcileCron.ts`:
   - Internal scheduler (replace standalone process).
   - Per-adapter mutex to avoid overlapping runs.
   - Configurable interval via `RECONCILE_INTERVAL_MS`.
   - Emit reconcile duration and correction counts.

4. Add graceful shutdown handling:
   - stop routers/schedulers,
   - close all adapter connections intentionally,
   - await in-flight DB writes with timeout,
   - flush DLQ worker,
   - close Redis and logger cleanly.

---

### Phase 3 — DLQ + Health + Metrics

1. Add `src/workers/DLQWorker.ts`:
   - unified retry loop using Redis lists:
     - `dlq:{exchange}` -> `dlq:{exchange}:processing` -> `dlq:{exchange}:dead`.
   - max attempts from env (`DLQ_MAX_ATTEMPTS`).
   - retry interval via `DLQ_RETRY_INTERVAL_MS`.

2. Integrate DLQ writes in adapter DB write boundaries:
   - catch DB failures,
   - push serialized replay payload with handler metadata.

3. Add `src/workers/HealthServer.ts`:
   - `GET /health` aggregate manager + adapters + dlq sizes.
   - `GET /health/:exchange` details.
   - `GET /metrics` via existing `prom-client` registry.
   - bind `WORKER_HEALTH_PORT`.

4. Logging standardization:
   - ensure all adapter logs include structured context keys:
     - `exchange`, `userId` and where applicable `tradeId`, `orderId`, `event`, `latency_ms`.

---

### Phase 4 — PM2 and Runtime Cutover

1. Update `ecosystem.config.js`:
   - Replace 7 worker apps + `reconcile-cron` with single `worker-manager` app (`src/workers/index.ts`).
   - Keep API app unchanged.

2. Update package scripts/documentation:
   - Add worker start script (e.g. `bun src/workers/index.ts`),
   - update README operational commands.

3. Transitional rollout:
   - deploy manager with dual-read/dual-write compatibility enabled,
   - monitor health/metrics,
   - disable legacy workers,
   - disable compatibility mode,
   - remove legacy worker runner files after soak period.

## Verification Plan

1. **Unit/behavior checks**
   - command routing correctness by exchange,
   - intentional close does not reconnect,
   - reconnect backoff for unintentional disconnect,
   - health payload shape and per-adapter stats,
   - DLQ replay success/failure progression.

2. **Integration scenarios per exchange**
   - open market order -> trade reaches `waiting_targets` (or exchange-expected state),
   - close event -> trade becomes `closed` with pnl fields,
   - worker restart -> active users restored quickly,
   - simulated DB failure -> DLQ enqueue + retry + recover.

3. **Operational checks**
   - PM2 single worker process stable,
   - `/health` and `/metrics` reachable,
   - no duplicate processing during transition (dual mode).

---

## Risk & Mitigation

- **Risk:** Big-bang migration across 7 adapters introduces regressions.
  - **Mitigation:** compatibility mode, phased cutover, retain legacy workers until soak complete.

- **Risk:** Message loss/duplication when moving stream topology.
  - **Mitigation:** dual-write + explicit ACK/claim strategy; idempotent DB update guards retained.

- **Risk:** Reconcile overlap with event handlers.
  - **Mitigation:** per-adapter mutex and transaction boundaries.

---

## Definition of Done (updated for 7 exchanges)

- Single `bun src/workers/index.ts` manages all 7 exchange adapters.
- Executors send routable control commands with `exchange` (compatibility mode removed after cutover).
- Intentional close bug fixed across all adapters.
- `/health` returns aggregate + per-exchange connection/error/DLQ signals.
- DLQ retries failed writes and dead-letters after max attempts.
- PM2 runs API + single worker-manager process only.
- Legacy worker processes removed after successful soak window.
