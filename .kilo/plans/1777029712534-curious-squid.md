# Plan: Account Holdings & Balances Endpoint

## Goal
1. Add a `getBalances` method in every exchange service that hits the exchange API and returns the raw response (no DB writes, no normalization).
2. Add a unified router `/balances` that accepts an `exchange_id` query param, loads + decrypts credentials, dispatches to the right exchange service, and returns whatever the exchange API responded with.

Scope: futures account balances (primary use case per existing codebase — `market_type === 'futures'`). Spot handling can be layered later if needed; this pass returns futures balances where available, and spot where the exchange is clearly spot (Tokocrypto supports both).

---

## 1. Per-service `getBalances` methods

All methods assume `initialize(...)` has been called first (same pattern as every other service method). All return raw parsed JSON response. On HTTP failure return the existing error-shape `{ status: "error", message, statusCode }` convention already used in the codebase.

### `src/services/gateServices.ts`
Add `getBalances` after `getAccountInfo`. Hits **both** futures and spot in parallel and returns both payloads:
- Futures: `GET /api/v4/futures/usdt/accounts` (settle=usdt futures account)
- Spot: `GET /api/v4/spot/accounts` (list of per-currency spot balances)
- Both signed with `signRequestRestGate` (same pattern as `getAccountInfo`)
- Returned shape:
  ```ts
  { futures: <raw futures response>, spot: <raw spot response> }
  ```
- If one side errors, the error object is placed in its slot; the other side still returns. No aggregation/normalization.

### `src/services/okxServices.ts`
Add `getBalances`:
- Method: `GET`
- requestPath: `/api/v5/account/balance`
- Signed via `signRequestOkx` (same pattern as existing methods in that file)
- Returns raw parsed JSON

### `src/services/bitgetServices.ts`
A `fetchBalance` already exists (CCXT `fetchBalance()`). Add a thin alias `getBalances` that calls it, or keep both. Plan: add `getBalances` as the canonical name used by the router, which internally calls `this.exchange.fetchBalance()` with `{ type: 'swap' }` params to force futures wallet.

### `src/services/bitmartServices.ts`
Add `getBalances`:
- Uses CCXT: `this.exchange.fetchBalance({ type: 'swap' })`
- Return the raw CCXT result
- Standard try/catch returning the `{ status: 'error', ... }` shape on failure (matches file's style)

### `src/services/mexcServices.ts`
Add `getBalances`:
- Uses CCXT: `this.exchange.fetchBalance({ type: 'swap' })`
- Same pattern as bitmart

### `src/services/tokocryptoServices.ts`
Add `getBalances`:
- Uses CCXT: `this.exchange.fetchBalance()` — market type is already set at init (`spot` or `future`).
- Returns raw CCXT result

### `src/services/hyperliquidServices.ts`
`getClearinghouseState(userAddress)` already exists and is exactly the balance + positions endpoint. Add a thin convenience `getBalances(userAddress)` that delegates to `getClearinghouseState` so the router interface is uniform across all exchanges.

(Note: Hyperliquid balance is a public `/info` endpoint that takes a wallet address — no signing needed. The router will pass `wallet_address` from unwrapped credentials.)

## 2. New router: `src/routes/balancesRoute.ts`

New file. Exposes a single endpoint:

```
GET /balances?exchange_id=123
```

### Flow
1. Validate `exchange_id` query param (must be positive integer) — return 400 otherwise.
2. Load exchange row: `postgresDb.query.exchanges.findFirst({ where: eq(exchanges.id, id) })`. 404 if not found.
3. Dispatch on `exchange.exchange_title`:
   - `gate`  → `GateHandler.unwrapCredentials` → `GateServices.initialize(key, secret)` → `GateServices.getBalances()`
   - `okx`   → `OkxHandler.unwrapCredentials` → `OkxServices.initialize(key, secret, passphrase)` → `OkxServices.getBalances()`
   - `bitget` → `BitgetHandler.unwrapCredentials` → `BitgetServices.initialize(key, secret, passphrase)` → `BitgetServices.getBalances()`
   - `bitmart` → `BitmartHandler.unwrapCredentials` → `BitmartServices.initialize(key, secret, memo)` → `BitmartServices.getBalances()`
   - `mexc` → `MexcHandler.unwrapCredentials` → `MexcServices.initialize(key, secret)` → `MexcServices.getBalances()`
   - `tokocrypto` → `TokocryptoHandler.unwrapCredentials` → `TokocryptoServices.initialize(key, secret, marketType)` → `TokocryptoServices.getBalances()`
   - `hyperliquid` → `HyperliquidHandler.unwrapCredentials` → `HyperliquidServices.getBalances(wallet_address)` (no initialize needed for info endpoint)
   - default → 400 `Unsupported exchange`
4. `try/finally`: always call the service's `clearCredentials()` in `finally` (mirrors existing handler patterns like `futuresOrderDb`). Hyperliquid has no creds to clear for info; still safe to call.
5. Response shape:
   ```json
   {
     "exchange_id": 123,
     "exchange_title": "gate",
     "data": <raw exchange response>
   }
   ```
   On exchange-side error (status: "error" from service), forward the error object with its `statusCode`.
6. Wrap everything in try/catch and log via `createLogger({ process: 'balances' })` — matches existing logger usage.

### Register in `src/index.ts`
Add:
```ts
import balancesRouter from './routes/balancesRoute'
...
app.route('/balances', balancesRouter)
```

## 4. Non-goals (for this pass)
- No scheduled/cron daily pull. Router is just the on-demand endpoint the daily worker will call.
- No persistence / no writes to `user_balances_snapshots`.
- No normalization across exchanges — each response shape is preserved verbatim.
- No auth/middleware additions — inherits whatever the rest of the app uses (none currently on most routes).

---

## 5. Open questions / assumptions
- **Futures vs spot:** Gate returns both futures and spot. Others (okx/bitget/bitmart/mexc) stay futures-only for this pass. Tokocrypto honors whatever `market_type` was stored.
- **Hyperliquid spot:** Only `clearinghouseState` (perps) is returned. Can add `spotClearinghouseState` later if needed.
- **Response envelope:** Proposed `{ exchange_id, exchange_title, data }`. If you prefer returning the raw exchange body at the top level, easy change.
